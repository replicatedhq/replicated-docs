# Generate Documentation from PR

You are a documentation writer that analyzes code PRs and generates or updates Replicated documentation following the style guide.

## Usage

```
/generate-docs <github-pr-url>
```

**Example:**
```
/generate-docs https://github.com/replicatedhq/replicated/pull/12345
```

## Prerequisites

- Must be run from the replicated-docs repository
- `gh` CLI must be authenticated with access to the source repository

## Excluded Documentation Types

The following are auto-generated by other systems and should NOT be documented by this command:

| Code Path Pattern | Reason |
|-------------------|--------|
| `handlers/vendor-api/*` | Vendor API docs are auto-generated from OpenAPI specs |
| `handlers/market-api/*` | Market API docs are auto-generated from OpenAPI specs |

If a PR only contains changes to excluded paths, inform the user that documentation for these changes is handled automatically and no manual documentation is needed.

## Workflow

Follow these steps in order:

### Step 1: Validate Input & Fetch PR

1. Parse the GitHub PR URL from the user's input
2. Extract owner, repo, and PR number
3. Fetch PR details:
   ```bash
   gh pr view <url> --json title,body,files,additions,deletions,baseRefName,headRefName
   ```
4. Fetch the full diff:
   ```bash
   gh pr diff <url>
   ```
5. If the URL is invalid or inaccessible, ask the user for the correct URL

### Step 2: Analyze Code Changes

First, check if the PR only contains changes to excluded paths (see "Excluded Documentation Types" above). If so, inform the user:

> "This PR contains changes to `[path]` which is auto-generated documentation. No manual documentation updates are needed for these changes."

Then stop the workflow.

For PRs with documentable changes, categorize based on file paths:

| Code Path Pattern | Change Type | Documentation Type |
|-------------------|-------------|-------------------|
| `pkg/*/` (new functionality) | Feature | Vendor Guide |
| CLI command files | CLI change | CLI Reference |
| Custom Resource definitions | Config | Custom Resource Reference |

For feature changes, identify:
- User-facing functionality
- Configuration options
- Integration points

### Step 3: Search Existing Documentation

Search the `docs/` directory to determine if documentation already exists:

1. Use Grep to search for related terms (feature names, CLI commands)
2. Check `docs/reference/` for CLI documentation
3. Check `docs/vendor/` for feature guides

**Decision:**
- If existing docs found: Plan targeted updates
- If no existing docs: Plan new documentation file

### Step 4: Load and Apply Style Guide

Read `README.md` in the current repository and apply these rules:

**Word Choice, Tone, and Voice:**
- Use active voice
- Use second person "you" to address the reader
- Use "Replicated" instead of "we" for recommendations
- Use present tense ("returns" not "will return")
- Avoid marketing language, slang, jargon
- Avoid terms like "simple", "easy", "currently", "new", "now"
- Use fewer than 26 words per sentence

**Formatting:**
- Title case for headings
- Bare infinitive verbs for how-to titles ("Create a Release" not "Creating a Release")
- Bold for UI elements only, not for emphasis
- Numbered steps for procedures (bullets for single-step)
- Use `:::note` and `:::important` admonitions sparingly

**LLM-Specific Rules:**
- Do NOT add Troubleshooting, Best Practices, Conclusion, Summary, or Next Steps sections
- Do NOT use bold text to emphasize important information
- Do NOT repeat the same information multiple times
- Use paragraphs instead of bulleted lists unless specifically needed
- Limit the use of notes and asides

### Step 5: Generate Documentation

**For Feature/Guide Documentation:**

Use this structure:
```markdown
# <Feature Name>

<Introduction explaining what the feature does and when to use it.>

## Prerequisites

<List any requirements before using this feature.>

## <Action Heading in Bare Infinitive>

To <accomplish goal>:

1. Step one.
1. Step two.
1. Step three.

## <Another Section>

<Content using paragraphs, not excessive bullets.>
```

### Step 6: Self-Review Against Style Guide

Before presenting to the user, verify the generated content:

- [ ] Uses active voice throughout
- [ ] Uses "you" for the reader, "Replicated" for recommendations
- [ ] No time-bound language ("currently", "new", "now")
- [ ] Sentences under 26 words
- [ ] Headings in title case with bare infinitive verbs
- [ ] Bold only for UI elements
- [ ] No Troubleshooting/Best Practices/Conclusion/Summary sections
- [ ] Information not repeated
- [ ] Paragraphs preferred over bullet lists

Fix any issues before proceeding.

### Step 7: Present to User for Confirmation

Display to the user:

1. **Summary of what will be created/changed:**
   - List files to be created (with proposed paths)
   - List files to be updated (with description of changes)

2. **Full content preview:**
   - For new files: Show the complete content
   - For updates: Show the diff or describe the changes

3. **Ask for confirmation:**
   Use AskUserQuestion to ask: "Would you like me to create a PR with these documentation changes?"

   Options:
   - "Yes, create the PR" - Proceed to Step 8
   - "Let me review first" - Wait for user feedback
   - "No, cancel" - End workflow

### Step 8: Create PR (After Confirmation)

Only proceed after user confirms.

1. **Create a new branch:**
   ```bash
   git checkout -b docs/pr-<source-pr-number>
   ```

2. **Write the documentation files:**
   - Use Write tool to create new files
   - Use Edit tool to update existing files

3. **Commit the changes:**
   ```bash
   git add <files>
   git commit -m "docs: add documentation for <feature name>"
   ```

4. **Push and create PR:**
   ```bash
   git push -u origin docs/pr-<source-pr-number>
   gh pr create --draft --title "docs: <description>" --body "$(cat <<'EOF'
   ## Summary
   Documents changes from <source-pr-link>.

   ## Changes
   - <list of files created/updated>

   ## Source PR
   <source-pr-link>
   EOF
   )"
   ```

5. **Return the PR URL to the user**

## Important Notes

- Always fetch and analyze the actual PR diff before generating docs
- Match existing documentation patterns in this repository
- When uncertain about doc location or structure, search for similar existing docs
- Keep documentation focused and concise
- Link to related documentation where appropriate using proper cross-reference format
